# Llm Specification

> Generated by openlore v1.0.0 on 2026-03-01 16:52
> Source files: src/core/services/llm-service.ts

## Purpose

Provides a unified interface for interacting with various LLM (Large Language Model) providers,
handling authentication, request formatting, error parsing, and response normalization.

## Entities

### LLMProvider

Represents a Large Language Model provider that can generate completions and count tokens. This is
an interface that concrete providers implement.

**Properties:**

| Name | Type | Description |
|------|------|-------------|
| name | string | The name of the LLM provider (e.g., 'anthropic', 'openai'). |
| maxContextTokens | number | The maximum number of tokens the provider can handle in a single context window. |
| maxOutputTokens | number | The maximum number of tokens the provider can generate in a single response. |

### CompletionRequest

Represents a request to generate a completion from an LLM provider.

**Properties:**

| Name | Type | Description |
|------|------|-------------|
| systemPrompt | string | The system-level prompt that sets the context or behavior for the LLM. |
| userPrompt | string | The user-provided prompt for which the LLM generates a completion. |
| temperature | number | Controls the randomness of the LLM's output. Lower values make the output more deterministic. |
| maxTokens | number | The maximum number of tokens the LLM should generate in the response. |
| stopSequences | string[] | Sequences that signal the LLM to stop generating further tokens. |
| responseFormat | 'text' | 'json' | The format of the LLM's response (e.g., plain text or JSON). |

### CompletionResponse

Represents the response generated by an LLM provider for a completion request.

**Properties:**

| Name | Type | Description |
|------|------|-------------|
| content | string | The generated content from the LLM. |
| usage | object | Token usage statistics for the completion request. |
| model | string | The model used to generate the completion. |
| finishReason | 'stop' | 'length' | 'error' | The reason the LLM stopped generating tokens (e.g., stop sequence, max length, or error). |

### TokenUsage

Tracks token usage across multiple LLM requests for monitoring and cost estimation.

**Properties:**

| Name | Type | Description |
|------|------|-------------|
| inputTokens | number | Total number of input tokens used across all requests. |
| outputTokens | number | Total number of output tokens generated across all requests. |
| totalTokens | number | Total number of tokens used (input + output) across all requests. |
| requests | number | Total number of requests made. |

### CostTracking

Tracks the estimated cost of LLM usage across providers for budgeting and monitoring.

**Properties:**

| Name | Type | Description |
|------|------|-------------|
| estimatedCost | number | The total estimated cost in the specified currency. |
| currency | string | The currency used for cost estimation (e.g., 'USD'). |
| byProvider | Record<string, number> | Breakdown of estimated costs by provider. |

### OpenAIProvider

A concrete implementation of the LLMProvider interface for interacting with OpenAI's language
models.

**Properties:**

| Name | Type | Description |
|------|------|-------------|
| apiKey | string | The API key used to authenticate requests to the OpenAI API. |
| model | string | The specific OpenAI model to use for text completion (e.g., 'gpt-4o'). |
| baseUrl | string | The base URL for the OpenAI API endpoint. |

**Relationships:**

- implements LLMProvider (OpenAIProvider implements the LLMProvider interface.)

## Requirements

### Requirement: LLMProviderValidation

The system SHALL validate LLMProvider according to these rules:
- Provider must implement the `generateCompletion` method to return a `CompletionResponse`.
- Provider must implement the `countTokens` method to return a token count as a number.

#### Scenario: GenerateCompletionWithValidRequest
- **GIVEN** A valid `CompletionRequest` and a configured LLM provider
- **WHEN** The `generateCompletion` method is called
- **THEN** A `CompletionResponse` is returned with the generated content and token usage

#### Scenario: CountTokensForGivenText
- **GIVEN** A text input and a configured LLM provider
- **WHEN** The `countTokens` method is called
- **THEN** The token count for the input text is returned as a number

### Requirement: CompletionRequestValidation

The system SHALL validate CompletionRequest according to these rules:
- System prompt and user prompt must be non-empty strings.
- Temperature must be a number between 0 and 1 if provided.
- Max tokens must be a positive integer if provided.

#### Scenario: GenerateTextCompletion
- **GIVEN** A valid `CompletionRequest` with `responseFormat` set to 'text'
- **WHEN** The LLM provider processes the request
- **THEN** A text-based completion is returned in the `CompletionResponse`

#### Scenario: GenerateJsonCompletion
- **GIVEN** A valid `CompletionRequest` with `responseFormat` set to 'json'
- **WHEN** The LLM provider processes the request
- **THEN** A JSON-based completion is returned in the `CompletionResponse`

### Requirement: CompletionResponseValidation

The system SHALL validate CompletionResponse according to these rules:
- Content must be a non-empty string.
- Input, output, and total tokens must be non-negative integers.
- Model must be a non-empty string.

#### Scenario: CompletionStopsDueToStopSequence
- **GIVEN** A `CompletionRequest` with a valid stop sequence
- **WHEN** The LLM encounters the stop sequence during generation
- **THEN** The `finishReason` in the `CompletionResponse` is set to 'stop'

#### Scenario: CompletionStopsDueToMaxTokens
- **GIVEN** A `CompletionRequest` with a `maxTokens` limit
- **WHEN** The LLM reaches the `maxTokens` limit during generation
- **THEN** The `finishReason` in the `CompletionResponse` is set to 'length'

### Requirement: TokenUsageValidation

The system SHALL validate TokenUsage according to these rules:
- Input, output, and total tokens must be non-negative integers.
- Requests must be a non-negative integer.

#### Scenario: TrackTokenUsageForASingleRequest
- **GIVEN** A `CompletionResponse` with token usage data
- **WHEN** The token usage is recorded
- **THEN** The `TokenUsage` object is updated with the new token counts and request count

#### Scenario: AggregateTokenUsageAcrossMultipleRequests
- **GIVEN** Multiple `CompletionResponse` objects with token usage data
- **WHEN** The token usage is recorded for each request
- **THEN** The `TokenUsage` object aggregates the token counts and increments the request count

### Requirement: CostTrackingValidation

The system SHALL validate CostTracking according to these rules:
- Estimated cost must be a non-negative number.
- Currency must be a non-empty string.
- Provider costs in `byProvider` must be non-negative numbers.

#### Scenario: ValidCostTrackingAccepted
- **GIVEN** A valid CostTracking object with all required fields
- **WHEN** The object is validated
- **THEN** Validation passes with no errors

### Requirement: OpenAIProviderValidation

The system SHALL validate OpenAIProvider according to these rules:
- apiKey must be a non-empty string
- model must be a non-empty string
- baseUrl must be a valid URL

#### Scenario: SuccessfulOpenaiApiRequest
- **GIVEN** A valid CompletionRequest and valid API credentials
- **WHEN** The generateCompletion method is called
- **THEN** A CompletionResponse is returned with the generated content from OpenAI

#### Scenario: ApiRequestFailureDueToInvalidApiKey
- **GIVEN** A CompletionRequest and an invalid API key
- **WHEN** The generateCompletion method is called
- **THEN** An error is thrown with status 401 (Unauthorized)

#### Scenario: RateLimitExceeded
- **GIVEN** A CompletionRequest and a rate-limited API key
- **WHEN** The generateCompletion method is called
- **THEN** An error is thrown with status 429 (Too Many Requests) and retryable flag set to true

### Requirement: Generatecompletion

The system SHALL generate a text completion from an llm provider based on the provided request, including system/user prompts, temperature, and stop sequences.

#### Scenario: Unnamed
- **GIVEN** A valid CompletionRequest with systemPrompt='You are a helpful assistant' and userPrompt='What is
the capital of France?', and the LLM API returns a successful response.
- **WHEN** generateCompletion is called with the request.
- **THEN** A CompletionResponse is returned with content='Paris', non-zero token usage, and
finishReason='stop'.

### Requirement: Parseerror

The system SHALL parse llm api errors into structured error objects with status codes and retryable flags.

#### Scenario: Unnamed
- **GIVEN** An error message 'Rate limited' and status code 429.
- **WHEN** parseError is called with the error and status.
- **THEN** An Error object is returned with status=429, message='Rate limited', and retryable=true.

### Requirement: Counttokens

The system SHALL estimate the number of tokens in a given text string for the llm provider.

#### Scenario: Unnamed
- **GIVEN** A non-empty string 'Hello, world!'.
- **WHEN** countTokens is called with the text.
- **THEN** A number greater than zero is returned.

### Requirement: OpenaiGeneratecompletion

The system SHALL generate a text completion using openai's api based on the provided completionrequest.

#### Scenario: Unnamed
- **GIVEN** A valid CompletionRequest with systemPrompt='You are a helpful assistant' and userPrompt='What is
2+2?', and OpenAI API returns a successful response.
- **WHEN** generateCompletion is called with the request.
- **THEN** A CompletionResponse is returned with content='4', non-zero token usage, and finishReason='stop'.

### Requirement: OpenaiCounttokens

The system SHALL estimate the number of tokens in a given text string using openai's tokenization rules.

#### Scenario: Unnamed
- **GIVEN** A string 'Hello, world!' and OpenAI's tokenization rules.
- **WHEN** countTokens is called with the text.
- **THEN** A number matching OpenAI's token count for the text is returned.

- **THEN** An OpenAIProvider instance is created with apiKey='sk-123', model='gpt-4o', and
baseUrl='https://api.openai.com/v1'.

### Requirement: CursorAgentCliProvider

The system SHALL provide an LLM provider identified as `cursor-agent` that implements `LLMProvider`, invokes the Cursor Agent CLI in non-interactive print mode to obtain completions, and returns a normalized `CompletionResponse`.

The implementation SHALL:

- Combine `systemPrompt` and `userPrompt` into a single prompt when both are present (consistent with other CLI providers in this codebase).
- Resolve the CLI executable path from an environment variable (e.g. override for non-standard installs) with a documented default binary name.
- Pass arguments appropriate for structured output (e.g. print mode and JSON output format) so the stdout can be parsed deterministically when the CLI supports it.
- On CLI failure (non-zero exit, stderr/stdout error detail), throw an error marked non-retryable where that pattern is used for other CLI providers.
- When token usage is not present in CLI output, SHALL derive input/output token estimates using the same estimation approach as other CLI providers.

#### Scenario: SuccessfulCompletion

- **WHEN** a valid `CompletionRequest` is provided and the Cursor Agent CLI returns successful structured output containing the assistant text
- **THEN** `generateCompletion` returns a `CompletionResponse` whose `content` is that text and whose `usage` contains non-negative token counts and a `totalTokens` equal to the sum of input and output tokens

#### Scenario: CliProcessFailure

- **WHEN** the Cursor Agent CLI exits with an error or returns output that cannot be parsed as expected
- **THEN** `generateCompletion` throws an error that surfaces CLI diagnostics and does not return a successful `CompletionResponse`

### Requirement: CursorAgentProviderRegistration

The system SHALL register the `cursor-agent` provider in the LLM service factory such that `createLLMService({ provider: 'cursor-agent', ... })` instantiates the Cursor Agent CLI-backed provider without requiring `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, `OPENAI_COMPAT_API_KEY`, or `GEMINI_API_KEY`.

The system SHALL include `cursor-agent` in provider pricing lookup with zero per-token estimated cost for cost estimation purposes (same class as other subscription or local CLI providers).

#### Scenario: FactorySelectsCursorAgent

- **WHEN** `createLLMService` is called with `provider: 'cursor-agent'`
- **THEN** the underlying provider’s `name` is `cursor-agent` and no cloud API key environment variable is read for instantiation

#### Scenario: CostLookupCliProvider

- **WHEN** `lookupPricing` is called for provider `cursor-agent` and any model identifier
- **THEN** returned input and output per-million token rates are zero

### Requirement: OpenAI structured output schema normalization

The system SHALL normalize JSON schemas sent through OpenAI and OpenAI-compatible `response_format.json_schema.schema` so that every object schema is closed, every object property is required, and properties that were optional in the caller schema remain nullable in the provider schema.

#### Scenario: OpenAI object schemas are closed
- **WHEN** `OpenAIProvider` sends a JSON schema with an object schema through `response_format.json_schema.schema`
- **THEN** the provider schema object has `additionalProperties: false`

#### Scenario: OpenAI object required includes all properties
- **WHEN** `OpenAIProvider` sends a JSON schema with object `properties`
- **THEN** the provider schema object's `required` array contains every property key

#### Scenario: Optional OpenAI properties become nullable
- **WHEN** `OpenAIProvider` sends a JSON schema where an object property is absent from the caller schema's original `required` array
- **THEN** the corresponding provider schema property allows `null`

#### Scenario: Nested OpenAI object schemas are normalized
- **WHEN** `OpenAIProvider` sends a JSON schema containing nested object schemas
- **THEN** each nested object schema has `additionalProperties: false` and a `required` array containing every nested property key

#### Scenario: Top-level OpenAI array schemas are wrapped as closed objects
- **WHEN** `OpenAIProvider` sends a top-level array JSON schema through `response_format.json_schema.schema`
- **THEN** the provider schema root is an object with an `items` property, `required` contains `items`, and the root has `additionalProperties: false`

#### Scenario: Caller schemas are not mutated
- **WHEN** OpenAI schema normalization prepares a provider schema
- **THEN** the caller-provided JSON schema remains unchanged

#### Scenario: OpenAI-compatible schemas use the same normalization
- **WHEN** `OpenAICompatibleProvider` sends a JSON schema through `response_format.json_schema.schema`
- **THEN** the provider schema follows the same object closure, required-property, nullable optional-property, and top-level array wrapper rules as `OpenAIProvider`

#### Scenario: Disabled OpenAI-compatible response format remains omitted
- **WHEN** `OpenAICompatibleProvider` is configured with `disableResponseFormat=true`
- **THEN** the request body does not include `response_format`
