# Agent Token-Efficiency Benchmark (WITH vs WITHOUT openlore)

> Spec 14. Does configuring openlore make a headless agent cheaper / fewer round-trips at
> answering relational questions about a codebase? This is the end-to-end **agent** story,
> distinct from [scripts/BENCHMARKS.md](../scripts/BENCHMARKS.md) (raw query **latency**, orient ~430µs p50).
>
> The table below the marker is regenerated by `npm run bench:agent`; this findings section is
> hand-maintained and preserved across re-runs.

## Honesty contract (Spec 25)

The README [Value Scorecard](../README.md#value-scorecard--does-it-pay-for-itself) and every public
token/cost claim about OpenLore are bound by these rules:

1. **No unmeasured numbers.** Never publish a savings figure the `bench:agent` harness did not
   produce. Intuitions/estimates are labeled as such or deleted — not dressed up as results.
2. **Always show the losses.** The negative cases (small/familiar repos: +43% cost) are published
   next to the wins, in the same table. Suppressing a loss cell is a bug.
3. **Date-stamped and re-measured.** The scorecard carries the measurement date and pinned SHAs, and
   is re-run after each optimization phase (Spec 25 Phases B–D). The README number is always the last
   measured number.
4. **Reproducible by anyone.** Every claim traces to a command in this repo (`npm run bench:agent`,
   and the forthcoming `openlore prove` for your-own-repo numbers). If it doesn't reproduce, it's
   marketing, and we remove it.

## Findings — first run (2026-06-01, sonnet, 4 runs/task, 5 repos)

**On this repo set, openlore did NOT reduce cost or tokens — it added overhead.** Across the
relational tasks the median was **+43% cost, +79% fresh input tokens, and +38% round-trips WITH
openlore**, and it lost on 5 of 7 tasks (break-even on the other 2). **Correctness was 100% in both
conditions on every task** — so openlore bought no accuracy either. This is reported as-is, per the
Spec 13 "report losses honestly; do not bury this number" discipline.

**Why — and why this is the wrong arena for the claim, not a refutation of it.** The headline claim
is that orientation burns *15,000–50,000 tokens* of file-by-file rediscovery that openlore replaces.
That premise **did not hold for these repos**: the WITHOUT baseline answered correctly using only
**~1,800–6,000 fresh tokens**. The reason is structural, not incidental — every pinned repo
(chalk / express / flask / gin / zod) is small, canonical, and **heavily represented in the model's
training data**. The model already "knows" express's router and Flask's request lifecycle, so a
single grep confirms what it half-remembers; there is no orientation tax to eliminate. Against that,
openlore's fixed costs — ~45 MCP tool definitions in every system prompt, an `orient()` round-trip,
and the agent still spot-reading to verify — are pure overhead. **openlore is designed for the
opposite case: large, unfamiliar, private codebases the model has never seen, where blind
exploration genuinely explodes.** None of the pinned repos exercise that case.

**What this means (the kill-signal, read precisely).** Spec 13 names this benchmark the earliest
kill-signal. It has fired — but it falsifies the claim *only on popular small repos*, which is the
benchmark's blind spot, not openlore's target. Two honest conclusions:

1. **The README token-savings figure is not supported by any measurement we have, and is
   contradicted on popular repos.** It has been corrected to say so (no longer a quiet "hypothesis").
2. **The decisive test was deferred to Round 2 (below), which DID run it** — large repos + deep
   multi-hop traces — and found openlore *does* help there once the tool surface is right. So the
   Round-1 negative is real but **scope-limited to small/familiar repos + shallow queries**, not a
   verdict on the idea. Read Round 2 for the measured win.

**Caveats on the numbers themselves.** Cost is noisy because Anthropic's prompt cache warms across
back-to-back runs (later runs of a repo are cheaper); `fresh input tokens` is the cache-robust metric
and tells the same story. Whether the WITH agent actually called `orient()` each time is not captured
by `--output-format json` (a stream-json pass to log tool calls is a worthwhile follow-up). N=4 is the
spec floor; variance ([min–max]) is shown per cell.

## Round 2 — research-driven fixes + the right arena (2026-06-01) — openlore wins on deep tasks

After Round 1, I studied how comparable tools benchmark and where they win (CodeGraph publishes
35% cheaper / 57% fewer tokens / 62% fewer tool calls; Serena reports 70–80% via LSP semantic chunks).
Three concrete fixes followed, all now in the harness:

1. **Right arena.** Added large repos + **deep multi-hop trace** questions mirroring CodeGraph's exact
   set (Django ORM query execution, Tokio scheduler, Excalidraw render, OkHttp interceptor chain, Gin
   middleware) — "how does X flow through Y", not the shallow "who calls foo" of Round 1. On these, the
   baseline genuinely explodes (Django: ~18 tool-calls, ~12–24k fresh tokens), so there is real
   orientation cost to remove.
2. **Isolation.** Both arms now run `--strict-mcp-config` (a globally-registered openlore can no longer
   leak into the WITHOUT baseline) — matching CodeGraph's method.
3. **Tool surface.** WITH defaults to `--minimal` (5 tools) — the MCP best-practice that unused tool
   schemas are pure per-request overhead. `--with-full-tools` exposes all ~45 for comparison.

**The diagnosis that led to the fix.** Round-1's `--minimal` was mis-curated for navigation — orient +
search_code + **3 governance tools** (record_decision / detect_changes / check_spec_drift), omitting
exactly the graph-traversal tools a trace question needs. Full mode (~45 tools) *had* them and cut
round-trips, but its schema overhead erased the saving. So we built a **navigation preset** (`openlore
mcp --preset navigation`, 7 graph tools: orient, search_code, get_subgraph, trace_execution_path,
analyze_impact, suggest_insertion_points, get_function_skeleton — CodeGraph parity): the right tools
*without* the overhead.

**N=4 result on the deep-trace tasks (the decisive run — see the auto-generated table below).** With
the navigation preset, openlore **flips from a loss to a win**:

| Aggregate (5 deep traces, N=4 medians) | WITHOUT | WITH `--preset navigation` |
|---|---|---|
| Cost | $0.159 | **$0.147 (−7%)** |
| Round-trips | 17 | **13 (−26%)** |
| Fresh input tokens | 21,885 | 21,961 (≈flat) |
| Correctness | 100% | 100% |

**And the win scales with codebase size** — exactly the hypothesis: excalidraw (~640 files) **−21%**,
tokio (~790) **−21%**, okhttp **−13%**, django (~3k) **−7%**, gin (110 files, smallest) **+4%** (≈even).
The clearest, most consistent signal is **round-trips: −26% aggregate, fewer on every task** — the
agent reaches the answer in fewer tool calls because `orient`/`trace_execution_path` collapse the
discovery loop. The cost win is real but modest (the orient/trace context roughly offsets the saved
reads in raw token terms; the gain comes through fewer round-trips), and weaker than CodeGraph's
published 25–35% — a response-compaction pass (CodeGraph's "adaptive sizing" that collapses redundant
implementations to signatures) is the obvious next lever.

**Bottom line — the honest, measured two-tier verdict:**
- **Small, familiar repos + shallow queries:** openlore *adds* cost (Round 1: +43%). The model already
  knows the code; there's no orientation tax to remove and the MCP surface is pure overhead. Don't use
  it here.
- **Larger codebases + deep multi-hop questions (its actual target):** openlore with the navigation
  preset is a net win — **−7% cost, −26% round-trips at N=4, scaling with repo size**, no accuracy
  cost. The idea works *in its arena*; rigorous testing — not a first-pass guess — was needed to show
  it, and to surface the tool-curation fix that made it true.

## Round 3 — live re-prove on this machine (2026-06-03, Spec 25 Phase E)

Round 2 deferred the actual re-run as "API-budget-gated." Spec 25 Phase E rescinded that and ran the
benchmark live (sonnet = `claude-sonnet-4-6`, the new default; `--preset navigation`; --strict-mcp-config).
Goals: (1) confirm the deep win still holds after the Phase C `expand`/`tokenBudget` changes, and
(2) re-measure the small/familiar **loss** case the navigation preset was never tested on (Round 1's
+43% used the mis-curated `--minimal`, not navigation). Small N (1–2) — directional, not definitive.

| Task | Arena | Δcost | Round-trips wo→w | N |
|------|-------|-------|------------------|---|
| okhttp-interceptor-chain | large / deep | **−13%** | 14 → 10 | 2 |
| chalk-callers-applyStyle | small / shallow | **−32%** | 6 → 3 | 2 |
| express-callers-router-route | small / shallow | **+59%** | 4 → 6 | 2 |
| express-callers-router-route | small / shallow | +32% | 4 → 5 | 1 |

Plus a live `openlore prove` self-run on **this repo** (3 graph-derived shallow locate/caller/callee
tasks, N=2/arm): **+27% cost, round-trips flat, 67%↔67% correctness → verdict "doesn't help here."**

**What Round 3 establishes, honestly:**
1. **The deep-task win is real and reproducible.** okhttp came in at **−13%, exactly matching Round 2** —
   the strongest possible reproducibility signal, and proof the Phase C changes didn't regress it.
2. **The small/familiar-repo loss is NOT eliminated, and it's not even uniform — it's task-dependent
   and high-variance.** Same repo class, opposite outcomes: chalk **−32% (win)** vs express **+59%
   (loss)**. The express loss is a *forced* orient round-trip (4→6) on a question the model could
   already answer in 4 turns; chalk's win is orient collapsing real exploration (6→3). On shallow
   "who/where" questions there is often no orientation tax to amortise, so the orient call is pure
   overhead — exactly as the thesis predicts.
3. **`openlore prove` works and tells the truth.** On this repo's shallow tasks it honestly returns
   "doesn't help here." That *is* the value: rather than trusting our numbers, a user measures theirs.

**Verdict, refined and unchanged in spirit:** use OpenLore for **deep, multi-hop questions on larger /
unfamiliar codebases** (consistent, reproduced win). On **shallow questions or small familiar repos**
it ranges from a modest win to a real loss — so it's **break-even-to-negative there, and that's
fine**: `openlore prove` lets each user decide for their own repo + task mix. The §3 goal of *fully
eliminating* the small-repo loss via a leaner surface was **not achieved** — the navigation preset
does not close it, because the cost is the (sometimes redundant) orient round-trip, not tool-schema
bytes. We report that plainly rather than claim a fix we did not land.

### Methodology

- **Agent:** `claude -p --output-format json`, model `sonnet`, 4 run(s)/task, median reported.
- **Isolation:** both arms run with `--strict-mcp-config` so the agent uses ONLY the config we pass (a globally-registered openlore can't leak into the baseline). Same as CodeGraph's published benchmark.
- **Conditions:** WITHOUT = empty MCP config (grep/read tools only) — the baseline. WITH = openlore (the repo's **local build**): `openlore mcp --no-watch-auto --preset navigation` (**--preset navigation** — a lean graph-navigation surface, the MCP best-practice of not paying schema overhead for tools the agent never calls), repo pre-analyzed with `openlore analyze --no-embed`, **plus** a system-prompt instruction to call `orient()` before reading files (a faithful mirror of the shipped `openlore-orient` skill — measures the product, not tools the agent ignores).
- **Scoring:** correct = the agent's final answer contains every independently-verifiable expected substring (`expect.mustInclude` in `bench-agent.tasks.ts`), confirmed against the pinned source by grep — not derived from openlore's own graph.
- **Metrics:** **cost (USD)** is the bottom line (it prices fresh vs cached tokens correctly). Tokens are broken into *fresh* input (`input_tokens` + cache creation — what the model processed fresh) and *cached* reads (`cache_read_input_tokens` — ~10× cheaper); plus output, round-trips (`num_turns`), wall-clock.
